## 一、项目概览

| 项目 | 值 |
|------|-----|
| **Mod 名称** | VerminLordMod（蛊真人Mod） |
| **版本** | 1.1.0.4 |
| **作者** | 風笙咲 |
| **平台** | Terraria tModLoader (1.4.4, .NET 8.0) |
| **外部依赖** | SubworldLibrary, ParticleLibrary |
| **核心主题** | 蛊师世界模拟 — 黑暗森林、信息不透明、不可逆性、零和博弈、维度隔离 |
| **代码总量** | ~135,000 行 C#（1,170 个 .cs 文件） |
| **设计哲学** | ① 黑暗森林（任意角色可能背叛）② 信息不透明（NPC 信念为黑盒）③ 不可逆性（永久性世界改变）④ 零和博弈（资源稀缺）⑤ 维度隔离（空窍 vs 背包） |

## 五、v4.0 架构优化记录

### 5.1 WorldTimeHelper — 消除日计数冗余

**问题**：~15 个系统各自实现 `Main.GameUpdateCount / 36000` 日计数逻辑，存在不一致风险。
**方案**：提取 `WorldTimeHelper` 静态工具类，提供 `CurrentDay`、`IsNewDay()`、`DayProgress`、`DaysSince()` 四个统一 API。
**影响范围**：MortalLifeSystem、TerritoryControlSystem、CultivationLoopSystem、FactionWarSystem、WeatherSystem、AlchemySystem、GuBreedingSystem 等。

### 5.2 WeatherSystem + SeasonSystem 合并

**问题**：SeasonSystem 和 WeatherSystem 功能重叠（天气状态、修炼加成、视觉效果），可能导致状态冲突。
**方案**：将 Season、CelestialPhase、SeasonModifier 等合并到 WeatherSystem，SeasonSystem 标记 `[Obsolete]` 并代理到 WeatherSystem.Instance。
**向后兼容**：现有代码通过 SeasonSystem 访问仍可工作，但会收到编译器警告。

### 5.3 EventBus 内存泄漏修复

**问题**：EventBus 的订阅在 World Unload 时未清理，导致不同世界会话间的事件订阅残留。
**方案**：添加 `EventBusWorldHook` ModSystem，在 `OnWorldUnload()` 中调用 `EventBus.Clear()`。

### 5.4 GuClassificationSystem 自动注册

**问题**：GuClassificationSystem 的 `RegisterAllGus()` 为空 TODO，分类系统无法工作。
**方案**：实现自动扫描，遍历 `ItemLoader` 中所有 `IGu` ModItem 并自动注册分类条目。

### 5.5 ElementalTrailBehavior 泛型基类

**问题**：47 种 TrailBehavior 存在大量重复代码（TrailManager 管理、生命周期方法、属性代理）。
**方案**：创建 `ElementalTrailBehavior<TTrail>` 泛型基类，封装共享的 OnSpawn/Update/OnKill/PreDraw 逻辑，子类只需实现 `ConfigureTrail()` 方法配置特有属性。
**已迁移**：FireTrailBehavior、WaterTrailBehavior、IceTrailBehavior 已迁移到新基类。
**待迁移**：其余 44 种 TrailBehavior 可逐步迁移。

### 5.6 PNG 资源清理

**问题**：Systems 目录中存在未引用的 PNG 文件（MiZongBuff.png、MiZongZhenTile.png）。
**方案**：已删除未引用的 PNG 文件。DefenseMiZongBuff.png 和 DefenseMiZongZhenTile.png 已确认被 ModBuff/ModTile 引用，保留。

### 5.7 DefaultDaoHenMap 自动填充（★ v4.1）

**问题**：DefaultDaoHenMap 为空字典，道痕冲突系统无法实际运作。
**方案**：
1. 添加 `GuElementToDaoPath` 映射（GuElement → DaoPath），作为元素到道痕的桥梁
2. 实现 `PopulateDefaultDaoHenMap()` 方法，在 `OnWorldLoad()` 时自动扫描所有 ModItem
3. 推断优先级：`DaoWeapon.DaoType` → `InferDaoTypeFromHierarchy()`（类名/继承链）→ `GuElement` 映射
4. `GetDefaultDaoHenTag()` 调用前自动确保映射表已填充

**影响**：道痕冲突系统从 70% → 85% 完成度，所有 IGu 物品现已自动映射到 DaoPath。

### 5.8 GuClassificationSystem 双接口兼容（★ v4.1）

**问题**：`Abstractions.IGu`（含 GuElement/GuCategory/DaoType 属性）和 `Content.Items.IGu`（空标记接口）并存，原 RegisterAllGus() 仅识别前者，导致大多数蛊虫未被注册。
**方案**：修改 RegisterAllGus() 支持双接口：
- `Abstractions.IGu`：完整注册（元素/分类/道痕）
- `Content.Items.IGu`：基础注册（GuWeaponItem → Category=Attack, Level=GetGuLevel()；其他 → Category=Special, Level=1）

### 5.9 蛊虫掉落数据驱动重构（★ v4.1）

**问题**：GlobalNPCLoot.cs 含 300+ 行巨型 if-else 链，维护困难，容易产生重复掉落条目。
**方案**：创建 GuDropRegistry 数据驱动掉落系统，将所有掉落规则集中管理。

```
GuDropRegistry (静态类) — 数据驱动掉落注册表
├── GuDropEntry — 掉落条目（ItemType / ChanceDenominator / MinStack / MaxStack）
├── BossDropEntry — Boss 专属掉落（NPC TypeID + 掉落列表）
├── NpcGroupDrop — NPC 组掉落（NPC TypeID 列表 + 掉落列表）
├── UniversalDrops — 通用掉落池（DirtGu 1/200, ABitGu 1/2000, WineBug 1/20000）
├── BossCommonDrops — Boss 通用掉落池（WhitePig/BlackPig/BloodSkullGu 各 1/100）
├── BossDrops — Boss 专属掉落（按 NPC TypeID 索引）
├── EarlyBossGroupDrops — 早期 Boss 组掉落
├── SpecialNpcDrops — 特殊 NPC 掉落
├── PreHardmodeElementDrops — 困难模式前元素掉落
├── HardmodeElementDrops — 困难模式元素掉落
├── TownNpcDrops — 城镇 NPC 掉落
├── _npcDropCache — 掉落缓存（NPC TypeID → List<GuDropEntry>）
├── BuildCache() — 构建缓存（遍历所有组掉落，按 NPC TypeID 索引）
├── TryGetDrops() — 查询 NPC 掉落
├── TryGetBossDrops() — 查询 Boss 专属掉落
└── IsInEarlyBossGroup() — 判断是否属于早期 Boss 组
```

**GlobalNPCLoot 重构后**：
- 移除 300+ 行 if-else 链
- 首次调用时 BuildCache()
- Boss → BossCommonDrops + BossDrops
- EarlyBossGroup → EarlyBossGroupDrops
- 普通 NPC → TryGetDrops() 缓存查询
- Town NPC → TownNpcDrops
- 所有 NPC → UniversalDrops

**修复的掉落问题**：
- 移除 YuanS 直接掉落（按 D-32 决策，元石仅通过交易/资源节点获取）
- 移除重复 PotatoMotherGu 掉落条目
- 合并重复 JiaoLeiPotatoGu 掉落为单一 1/12 概率
